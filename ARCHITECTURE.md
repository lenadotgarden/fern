# Fern Architecture & Guidelines

Welcome to the Fern project! This document outlines the technical stack, architectural decisions, and design philosophy. Future AI agents should read this file before modifying the codebase.

## Tech Stack & Architecture

Fern is a cross-platform task manager mimicking the aesthetics and UX of **Things 3**.

### 1. Backend / Core Engine (`/core`)

- Written in **Rust**.
- Uses **SQLite** (via `rusqlite`) for data persistence.
- **Data Integrity**: The Rust core is responsible for data consistency. For example, if a `Task` belongs to a `Project`, the core automatically cascades the Project's `Area` to the Task.

**Modular Structure**:
- `src/models/`: Contains data structures (`area.rs`, `project.rs`, `task.rs`). Exposed via `src/models/mod.rs`.
- `src/database/`: Contains the DB interaction and queries.
  - `mod.rs`: Defines the `Database` struct and handles table initializations. It uses the `include!("...")` macro to import methods from repository files while keeping them as inherent methods on `Database` (e.g. `impl Database { ... }`).
  - `area_repo.rs`, `project_repo.rs`, `task_repo.rs`: Specific SQL queries and tests for each domain.
- `src/api/`: The main entry point API bridging the core logic to clients.
  - `mod.rs`: Defines the `FernAPI` struct (bridged by UniFFI). Includes modules via `include!("...")` macro so UniFFI detects all `#[uniffi::export]` blocks on `FernAPI`.
  - `area_api.rs`, `project_api.rs`, `task_api.rs`: Specific endpoints exposed to Swift.

### 2. Cross-Platform Bindings (`/core/src/api/mod.rs` -> UniFFI)

- We use **Mozilla's UniFFI** to generate Swift bindings from the Rust code.
- The bindings are built into an XCFramework (`apple/FernCore.xcframework`) via the script `apple/scripts/build_xcframework.sh`.
- **Important**: Whenever you modify the Rust core public models or methods in `api` or `uniffi::custom_type!`, you **MUST** re-run the build script (`cd apple/scripts && ./build_xcframework.sh`) before Xcode can see the changes.

### 3. Frontend Apple Client (`/apple`)

- Written in **SwiftUI**.
- Connected to the Rust core via `AppStore.swift` (`EnvironmentObject`), which holds the global state and calls `FernAPI` generated by UniFFI.
- **View Architecture (`apple/Shared/Views/`)**:
  - `Core/`: Foundational views (`MainContentView.swift`, `DropHandlers.swift`, `TrashView.swift`).
  - `Sidebar/`: Sidebar hierarchy and items (`SidebarItemView.swift`).
  - `Task/`: All task-related views (`InboxView.swift`, `TodayView.swift`, `TaskRowView.swift`, `TaskDetailView.swift`, etc.).
  - `Project/` & `Area/`: Detail views for Projects and Areas.
- **Important**: Whenever adding a new Swift file, it must be added to the `Fern` target in `apple/Fern.xcodeproj`. View files should **NOT** be added to the `FernTests` target (test targets use `@testable import Fern`).

### UI/UX Philosophy

- Matches **Things 3** clean and minimalist design.
- Avoid standard thick macOS `Form` or rounded gray backgrounds where possible. Use `.listStyle(.plain)`, unstyled `ScrollView`s with `VStack`, and native typography.
- Projects are nested under their Areas in the `SidebarView`.

## How to Add a Feature (TDD & Workflow)

We follow a strict Test-Driven Development (TDD) methodology. Here is the exact lifecycle of a new feature (e.g., adding "Tags" or "Checklists"):

### Step 1: Rust Models
1. Open the relevant file in `core/src/models/` (or create a new one).
2. Update the struct, constructors, and derive necessary traits (`uniffi::Record`, etc.).
3. Write unit tests for your model logic in the same file. Run `cargo test`.

### Step 2: Rust Database (Repositories)
1. Open the relevant file in `core/src/database/` (e.g., `task_repo.rs`).
2. Update SQL queries (Insert, Select, Update). If needed, update schema creation in `core/src/database/mod.rs`.
3. Add tests in the test block of the repository (e.g., `mod task_tests { ... }`).
4. Ensure robust handling of edge cases. Run `cargo test`.

### Step 3: Rust API (UniFFI Boundary)
1. Open the relevant file in `core/src/api/` (e.g., `task_api.rs`).
2. Expose the new methods on `impl FernAPI` with `#[uniffi::export]`.
3. Add end-to-end tests for the new API boundary in `core/src/api/mod.rs` (under `mod api_tests`). Run `cargo test`.

### Step 4: Rebuild Framework
1. Navigate to `apple/scripts/`.
2. Run `./build_xcframework.sh`.
3. Verify the script succeeds and updates `FernCore.xcframework`.

### Step 5: Swift AppStore (State Management)
1. Open `apple/Shared/AppStore.swift`.
2. Add `@Published` properties if new state tracking is needed.
3. Write the Swift wrapper methods that call the newly generated `store.api.yourNewMethod()`.
4. Update `apple/FernTests/AppStoreTests.swift` if there's complex logic. Run Xcode tests.

### Step 6: SwiftUI Views
1. Create or update views in `apple/Shared/Views/`.
2. Use `@EnvironmentObject var store: AppStore` to trigger the new actions.
3. Keep views small, modular, and in the correct domain folder (`Task/`, `Project/`, etc.).
4. Run the macOS build via Xcode or `xcodebuild` to ensure everything compiles seamlessly.
