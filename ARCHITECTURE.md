# Fern Architecture & Guidelines

Welcome to the Fern project! This document outlines the technical stack, architectural decisions, and design philosophy. Future AI agents should read this file before modifying the codebase.

## Tech Stack & Architecture
Fern is a cross-platform task manager mimicking the aesthetics and UX of **Things 3**.

1. **Backend / Core Engine (`/core`)**:
   - Written in **Rust**.
   - Uses **SQLite** (via `rusqlite`) for data persistence.
   - Core files: 
     - `src/models.rs`: Defines data structures (`Task`, `Project`, `Area`).
     - `src/database.rs`: DB interaction and queries.
     - `src/api.rs`: The main entry point API bridging the core logic to clients.
   - **Data Integrity**: The Rust core is responsible for data consistency. For example, if a `Task` belongs to a `Project`, the core automatically cascades the Project's `Area` to the Task.

2. **Cross-Platform Bindings (`/core/src/api.rs` -> UniFFI)**:
   - We use **Mozilla's UniFFI** to generate Swift bindings from the Rust code.
   - The bindings are built into an XCFramework (`apple/FernCore.xcframework`) via the script `apple/scripts/build_xcframework.sh`.
   - **Important**: Whenever you modify the Rust core public models or methods in `api.rs` or `uniffi::custom_type!`, you MUST re-run the build script (`cd apple/scripts && ./build_xcframework.sh`) before Xcode can see the changes.

3. **Frontend Apple Client (`/apple`)**:
   - Written in **SwiftUI**.
   - Connected to the Rust core via `AppStore.swift` (`EnvironmentObject`), which holds the global state and calls `FernApi` generated by UniFFI.
   - **UI/UX Philosophy**: 
     - Matches **Things 3** clean and minimalist design. 
     - Avoid standard thick macOS `Form` or rounded gray backgrounds. 
     - Use `.listStyle(.plain)`, unstyled `ScrollView`s with `VStack`, and native typography.
     - Projects are nested under their Areas in the `SidebarView`.

## Development Process (TDD)
We follow a strict Test-Driven Development (TDD) methodology.
1. When adding a new feature (e.g., tags, manual drag-and-drop order, checklists), **start in Rust**.
2. Write tests in `database.rs` or a new test file to validate the core logic.
3. Once the Rust core is robust and tested, expose it in `api.rs` and build the XCFramework.
4. Finally, implement the UI in SwiftUI (`ContentView.swift` and others).
